别急着堆 Agent：我最近对 OpenClaw 体系的三点收束

2026-03-22 · AI / Agent / OpenClaw

最近这轮折腾，我有一个很强的感觉：

很多 AI 系统不是能力不够，而是结构太松。

一开始很容易兴奋。看到一个新 skill，想接；看到一种多 agent 协作方式，想抄；看到一个自动化流程跑通了，就觉得体系快成了。

但跑几轮后会发现，真正让系统变乱的，通常不是“不会做”，而是这几个问题一直没写清楚：

  • 谁负责判断，谁负责执行
  • 哪些能力属于通用底座，哪些属于业务流程
  • 哪些动作可以自动下放，哪些必须卡住
  • Heartbeat 到底是在报平安，还是在推动系统进化

我最近把思路收回来，最后压成了三个判断。

1. 主 agent 不该什么都做，它该主要负责编排

这是这轮最重要的收束。

过去很容易默认一个强 agent 什么都能干：理解任务、查资料、写文案、改代码、发消息、更新文档、做巡检。短期看很爽，长期看一定失控。

因为一旦所有事都塞进主 agent，系统很快就会出现三个问题：

第一，判断和执行混在一起

主 agent 一边决定方向，一边亲自做大量体力活，最后最容易丢的是“边界判断”。

它本来应该问：

  • 这个任务该不该做
  • 该谁做
  • 风险在哪
  • 做完怎么验收

结果它花了大量时间在：看十篇资料、改三轮文案、手工跑流程、自己补各种细碎动作。主控一旦变重，编排能力就会变弱。

第二，子角色永远长不出来

如果主 agent 总能“顺手自己干完”，那研究、写作、运营、编程这些角色永远不会真的成型。

因为角色不是靠命名长出来的，是靠明确职责 + 可执行边界 + 稳定输入输出长出来的。

第三，错误会被一起放大

当“理解目标、选路径、做执行、发结果”都由一个角色包办时，任何一次判断偏差，都会直接流到外部动作上。

主 agent 的核心价值不是更能干，而是更会编排。

它应该主要负责：理解目标、任务拆解、角色调度、风险控制、最终验收，而不是默认亲自把所有活干完。

2. Skill 体系先别继续堆，先分三层

我后来越来越不喜欢“看到一个需求就加一个 skill”这种节奏。因为这样做，短期像在扩能力，长期其实是在扩混乱。

更稳的办法，是先把 skill 分成三层。

第一层：通用底座 / 基础能力

这一层解决的是“很多任务都会反复遇到”的共性问题，比如搜索、浏览网页、总结文档、读写文件、记忆检索、会话调度。它们不是某个业务独有的，而是很多业务都会反复调用的部件。

这一层最重要的不是“多”，而是“稳”。

第二层：业务能力

这一层才是面向具体结果的，比如飞书日历管理、飞书任务流、热点监控日报、文档生成与更新。用户真正要的，通常不是“调用一个 summarize”，而是“帮我把日报做出来”。

所以业务层的重点不是单工具，而是流程是否稳定、输入输出是否清楚、能否被编排复用。

第三层：社区扩展能力

这一层我现在会更克制。它适合放个性化很强的写作风格 skill、某个平台特化发布器、外部仓库迁来的实验型流程、不属于核心必需但值得尝试的能力。

这层不是不要，而是不能跟底座和业务混在一起。否则很快会出现一种假繁荣：skill 越来越多，但你越来越不知道哪个该给谁用，哪个只是实验品，哪个应该进主链路。

先分层，再扩张；先归类，再接入。

3. Heartbeat 最有价值的地方，不是提醒，是逼系统学习

以前看 Heartbeat，很容易把它理解成定时问一句还好吗、看下有没有新消息、查下任务有没有跑。这当然有用，但不够。

如果 Heartbeat 只负责“巡检”，它最多是个提醒器；如果它还能负责“学习 + 沉淀 + 验证”，它才开始像系统进化接口。

我现在更倾向于把 Heartbeat 固定成四步：

  • 先巡检：看关键流程有没有漏跑、异常、输出变差、该补跑的任务
  • 再学习：只看一小块和当前主线高度相关的输入
  • 必须沉淀：没有进入文档、模板、规则、记忆，就不算完成
  • 最后验证：这次调整怎么知道有效？下次用什么真实任务验证？两轮没反馈是不是该停掉？

很多“抽象优化”最容易自嗨的地方就在这：看起来想明白了，但系统没有任何变好证据。

Heartbeat 的价值，不是定时 ping，而是“巡检系统状态 + 推动结构进化的小闭环”。

最后，我现在对这套体系的判断

如果让我把最近的感悟压成最短的一版，就是下面三句：

  • 主 agent 要收回到“编排官”：最重要的工作不是多做，而是少做错事、少做重活、把边界守住。
  • Skill 要先分层，不要继续散装增长：底座、业务、扩展三层一旦不分，系统后面一定会越来越糊。
  • Heartbeat 要从“提醒器”升级成“学习闭环”：真正的价值不在于它多久响一次，而在于它有没有逼系统留下规则、模板和反馈。

我现在反而没那么急着继续加 agent 了。先把角色边界、skill 分层、调度矩阵和 heartbeat 模板跑顺，很多复杂度会自己降下来。

多 agent 从来不是自动增强器。基础没打稳时，它更像混乱倍增器。

先把结构搭稳，再谈扩张。
